# 練習 1：需求規範撰寫

## 目標
學習如何撰寫清晰、完整的應用程式需求規範，為 AI 生成高品質程式碼奠定基礎。

## 預計時間
30-40 分鐘

## 背景說明

優秀的需求規範是成功的關鍵。AI 的輸出品質直接取決於需求描述的清晰度和完整性。本練習將引導您撰寫專業級的需求文件。

## 任務：為線上書店系統撰寫需求規範

### 步驟 1：分析業務需求

請為一個線上書店系統撰寫需求規範。系統應該包含：
- 書籍瀏覽和搜尋
- 購物車功能
- 使用者帳戶管理
- 訂單處理

### 步驟 2：使用結構化模板

使用以下模板組織您的需求：

```markdown
# [應用程式名稱] 需求規範

## 1. 專案概述
### 1.1 目標
[描述專案的主要目標和價值主張]

### 1.2 目標使用者
[定義主要使用者群體]

### 1.3 核心功能
[列出 3-5 個最重要的功能]

## 2. 功能需求
### 2.1 [功能模組 1]
#### 使用者故事
作為 [使用者角色]，我想要 [功能]，以便 [價值]

#### 接受標準
- [ ] 標準 1
- [ ] 標準 2
- [ ] 標準 3

#### 技術規格
[具體的技術實現要求]

### 2.2 [功能模組 2]
[重複上述結構]

## 3. 非功能需求
### 3.1 效能需求
[回應時間、併發數等]

### 3.2 安全需求
[認證、授權、資料保護]

### 3.3 可用性需求
[無障礙、多語言等]

## 4. 資料模型
[關鍵實體和關係]

## 5. 使用者介面需求
[UI/UX 指導原則]

## 6. 限制條件
[技術限制、預算限制等]
```

### 步驟 3：撰寫您的需求規範

現在，使用上述模板撰寫線上書店系統的需求規範。

#### 範例開始：

```markdown
# 線上書店系統需求規範

## 1. 專案概述
### 1.1 目標
建立一個現代化的線上書店平台，提供便捷的書籍購買體驗，支援個人化推薦和社群互動功能。

### 1.2 目標使用者
- 主要：18-45 歲的讀書愛好者
- 次要：教育機構採購人員
- 第三：獨立作者和出版社

### 1.3 核心功能
1. 智慧書籍搜尋和推薦
2. 完整的購物車和結帳流程
3. 使用者評論和評分系統
4. 個人化閱讀清單管理
5. 行動裝置優化體驗

## 2. 功能需求
### 2.1 書籍瀏覽與搜尋
#### 使用者故事
作為一個讀者，我想要快速找到感興趣的書籍，以便節省選書時間。

#### 接受標準
- [ ] 支援關鍵字、作者、ISBN 搜尋
- [ ] 提供進階篩選（價格、評分、出版日期）
- [ ] 顯示搜尋結果的相關性排序
- [ ] 支援搜尋歷史和儲存搜尋

#### 技術規格
- 使用 Elasticsearch 實現全文搜尋
- 實施搜尋結果快取機制
- 支援模糊搜尋和自動完成
- 搜尋回應時間 < 500ms

[請繼續完成剩餘部分...]
```

### 步驟 4：加入雙語策略元素

在您的需求規範中加入雙語策略指示：

```markdown
## 實作指導

### 技術規格（英文）
[English Technical Specification]
- Use React 18 with TypeScript
- Implement Redux Toolkit for state management
- Use REST API with JWT authentication
- Database: PostgreSQL with Redis cache
- Deploy on AWS with auto-scaling

### 本地化需求（中文）
[Chinese Localization Requirements]
- 所有使用者介面使用繁體中文
- 價格顯示為新台幣（NT$）
- 日期格式：YYYY年MM月DD日
- 提供友善的錯誤訊息
- 支援繁體中文書名搜尋
```

## 實作任務

### 任務 A：完整需求文件
完成線上書店系統的完整需求規範，包含：
- [ ] 至少 5 個功能模組
- [ ] 每個模組的使用者故事
- [ ] 技術規格和限制
- [ ] 資料模型設計
- [ ] UI/UX 指導原則

### 任務 B：比較不同層次的需求
撰寫三個版本的需求：
1. **簡略版**（100-200 字）
2. **標準版**（500-800 字）
3. **詳細版**（1500+ 字）

比較三個版本生成的程式碼品質差異。

### 任務 C：領域特定需求
選擇一個特定領域，撰寫專業需求：
- 醫療系統（需符合 HIPAA）
- 金融系統（需符合 PCI DSS）
- 教育系統（需支援 SCORM）

## 評估標準

### 優秀需求規範的特徵
- **完整性**：涵蓋所有主要功能
- **清晰性**：無歧義的描述
- **可測試性**：明確的接受標準
- **技術可行性**：合理的技術要求
- **優先級**：功能有明確優先順序

### 常見問題檢查
- [ ] 是否有矛盾的需求？
- [ ] 是否遺漏重要功能？
- [ ] 技術規格是否過於限制？
- [ ] 是否考慮了邊界情況？
- [ ] 是否包含效能指標？

## 進階挑戰

### 挑戰 1：AI 對話式需求收集
設計一系列問題，引導 AI 幫助您完善需求：

```markdown
1. "What security considerations should I include for an e-commerce platform?"
2. "How should I structure the user authentication flow?"
3. "What are the best practices for shopping cart implementation?"
```

### 挑戰 2：需求優先級矩陣
使用 MoSCoW 方法分類需求：
- **Must have**：核心功能
- **Should have**：重要但非關鍵
- **Could have**：錦上添花
- **Won't have**：本期不實作

### 挑戰 3：需求驗證
請同學或 AI 審查您的需求，找出：
- 邏輯漏洞
- 技術衝突
- 遺漏場景
- 改進建議

## 反思問題

完成練習後，請思考：

1. **需求粒度**
   - 什麼程度的細節最適合 AI 理解？
   - 過於詳細會限制 AI 的創造性嗎？

2. **技術決策**
   - 應該在需求中指定技術棧嗎？
   - 如何平衡技術限制和靈活性？

3. **迭代改進**
   - 初始需求和最終需求差異多大？
   - 哪些部分需要多次調整？

## 提交作業

提交以下內容：
1. 完整的需求規範文件（Markdown 格式）
2. 三個不同詳細程度的版本
3. AI 生成程式碼的初步結果
4. 需求改進的迭代記錄
5. 反思問題的回答（300-500 字）
