# 產品需求文件 (PRD): Play right with AI Workshop

* **產品名稱:** Play right with AI: 一個以 GitHub Repo 為核心的 Playwright 實戰工作坊

* **作者:** Gemini

* **文件版本:** 3.0

* **日期:** 2025年9月11日

* **狀態:** 修訂中

## 1. 簡介與願景

### 1.1 文件目的

本文件旨在定義「Play right with AI」線上工作坊的產品願景、功能需求、目標受眾與成功指標，作為此 GitHub 教材開發與社群推廣的核心依據。

### 1.2 產品願景

我們的願景是打造一個開源、前瞻性的線上工作坊，核心是教授一種**「AI 驅動的自循環開發與測試流程」**。本工作坊以 GitHub Repo 為載體，引導學習者扮演「AI 指揮家」的角色，學習如何透過一系列精心設計的提示 (Prompts)，讓 AI 協同工作：**用 AI 生成應用程式碼，用 AI 分析程式碼並規劃測試，用 AI 編寫 Playwright 測試，再用 AI 分析測試結果並自我修復**。

工作坊特別強調**「英文思考，中文輸出」的進階提示詞策略**，利用 AI 在英文環境下的優越推理能力，產生更精確的技術內容，同時保持中文輸出以服務在地開發者。最終目標是賦能開發者與測試者，掌握這種高度自動化的「自舉式」工作流程。

### 1.3 問題陳述

生成式 AI 工具雖然強大，但多數開發者仍將其視為被動的「程式碼片段產生器」。他們缺乏一個系統性的方法論，來將多個 AI 工具鏈接成一個主動、閉環的工作流程。如何從一個高階的產品需求，透過 AI 協作，自動化地完成功能開發、測試規劃、腳本生成、執行、除錯，直到最終交付，是當前技術人員面臨的下一個效率革命的挑戰。

### 1.4 目標受眾

* **主要受眾:**

  * 擁抱 "Vibe Coding" 的前瞻性開發人員與測試工程師，他們不僅想用 AI 寫程式碼，更渴望學習如何**建構和指揮 AI 驅動的自動化工作流程**。

* **次要受眾:**

  * 希望將 AI 深度整合到團隊流程中的 DevOps 工程師。

  * 對下一代軟體開發與品質保證範式感興趣的技術主管與架構師。

### 1.5 成功指標

* **社群採納度:**

  * GitHub Repo 在發布後 6 個月內達到 500+ Stars。

  * GitHub Repo 在發布後 6 個月內被 Fork 100+ 次。

* **學習者參與度:**

  * 透過 Issue 或 Discussion 提出的有效問題與回饋超過 50 則。

  * 超過 100 名學習者完成最終的 Capstone 專案挑戰 (透過社群分享等方式追蹤)。

* **市場反應:**

  * 在技術部落格、社群媒體上獲得至少 10 次正面推薦或分享。

## 2. 工作坊架構與學習路徑

本工作坊的架構已完全重塑，以體現「AI 自循環」的核心理念。每個章節都是這個循環中的一個關鍵環節。

| 章節 | 標題 | 核心學習目標 | 
 | ----- | ----- | ----- | 
| **1** | **AI 指揮家：思維轉變與環境搭建** | 建立「AI 驅動自循環」的核心概念。一次性設定好所有工具：Playwright、VS Code、以及 Claude/Gemini CLI 等 AI 協作者。 | 
| **2** | **第一樂章：AI 生成應用程式** | 從一個簡單的自然語言需求開始（例如：「建立一個待辦事項應用」），指導 AI 生成一個基礎但功能完整的 Web 應用程式碼。**初步介紹「英文思考，中文輸出」策略的基礎應用。** | 
| **3** | **第二樂章：AI 擔任測試策略師** | 將上一章生成的應用程式碼交給 AI，並提示它：「作為一位資深 QA，請分析此應用並產出一份全面的 E2E 測試計畫。」 | 
| **4** | **第三樂章：AI 執行測試腳本編寫** | 使用 Playwright MCP，將 AI 生成的測試計畫轉化為指令，讓 AI 親自操作瀏覽器，並即時生成對應的 Playwright 測試腳本。 | 
| **5** | **第四樂章：AI 進行測試分析與除錯** | 故意在應用程式中引入一個 Bug。執行測試，然後將失敗的測試報告、Trace 檔案和相關程式碼餵給 AI，要求它診斷出問題的根本原因。 | 
| **6** | **最終樂章：AI 完成自我修復閉環** | 指示 AI 根據上一章的診斷結果，自動修復應用程式或測試腳本中的 Bug。重新執行測試，驗證修復是否成功，完成整個自循環。 | 
| **7** | **變奏曲：擴展與優化工作流** | 學習如何將此循環應用於更複雜的場景（例如，多頁面流程），以及如何優化提示詞 (Prompt) 來提高各環節 AI 輸出的品質，**包含「英文思考，中文輸出」的進階提示詞策略，以獲取更精確的技術性回覆**。 | 
| **8** | **Capstone 專案：指揮一場 AI 開發交響樂** | 學習者將收到一個全新的、高階的應用需求。任務是獨立運用所學，指揮 AI 完成從應用生成到測試修復的整個端對端自循環流程。 | 

## 3. 功能需求與交付項目

### 3.1 核心交付物：GitHub 專案庫

本工作坊的所有內容將存放在一個公開的 GitHub Repo 中，其結構如下：

* **`/prompts` 資料夾:**

  * **需求:** 存放每個章節核心任務的「黃金提示詞 (Golden Prompts)」。這是本工作坊最有價值的資產之一。**其中將包含運用「英文思考，中文輸出」策略的最佳實踐範例。**

* **`/workshop` 資料夾:**

  * **需求:** 包含每一章節的實作練習。每個練習的資料夾需包含：

    * `README.md`: 清晰的練習目標與步驟說明，引導學習者複製並使用 `/prompts` 中的提示詞。

    * `start-here/`: 供學習者開始練習的起始狀態（可能是一個需求文件或空的資料夾）。

    * `example-output/`: 該章節 AI 生成的程式碼、測試計畫或腳本的範例，供學習者參考。

* **根目錄 `README.md`:**

  * **需求:** 提供清晰的工作坊介紹、自循環流程圖、環境設定指南以及貢獻指南。

* **`/sample-app-source` 資料夾:**

  * **需求:** 存放一個預先準備好的範例網站，用於某些特定章節的練習，以確保學習體驗的一致性。

### 3.2 學習體驗需求

* **自我檢測:** 每個章節的 Markdown 文件結尾需包含「思考與挑戰」問題，例如「如何修改提示詞來處理圖片上傳的測試？」

* **互動式學習:** 鼓勵學習者透過 GitHub Issues 提問，並透過 GitHub Discussions 分享他們優化的提示詞或工作流。

* **雙語提示詞實踐:** 每個章節都包含「英文思考，中文輸出」的提示詞範例，展示如何：
  * 用英文構建邏輯推理部分
  * 指定中文輸出格式和術語
  * 比較純中文提示詞與雙語策略的效果差異

### 3.3 期末專案 (Capstone Project)

* **需求:** 在 `/workshop/08-capstone-project` 中提供一個綜合性挑戰。學習者只需得到一份 `需求.md` 文件，其餘所有工作（程式碼、測試、修復）都需透過指揮 AI 完成。

## 4. 非功能性需求

* **語言:** 所有教材、註解與文件均以繁體中文呈現。

* **文件品質:** Markdown 文件需格式清晰、易於閱讀，程式碼需有適當註解。

* **可維護性:** 專案結構需清晰，易於未來隨著 Playwright 或 AI 工具的更新而維護。提供貢獻指南，鼓勵社群共同維護。

## 5. 假設與依賴

* **假設:**

  * 學習者具備 JavaScript/TypeScript 的基礎程式設計能力。

  * 學習者了解 Web 開發的基本概念（HTML/CSS/DOM）。

  * 學習者擁有基本的命令列 (CLI) 與 Git 操作經驗。

  * **學習者擁有對應 AI 服務的 API 金鑰或訂閱（例如 OpenAI, Google AI, Anthropic）。**

* **依賴:**

  * 工作坊內容依賴於 Playwright 官方 API 的穩定性。

  * AI 相關章節依賴於第三方 AI 服務 (如 GitHub Copilot, Claude API, Gemini API) 的可用性與 Playwright MCP 等工具的穩定性。

## 6. 範圍以外 (Out of Scope)

* 深入的 JavaScript 或 TypeScript 程式語言教學。

* 提示工程 (Prompt Engineering) 的高深理論。

* 任何需要大量手動編寫程式碼的環節。
